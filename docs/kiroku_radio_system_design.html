Requirements

Before any architecture decision, we lock down what Kiroku must do (functional), what qualities it must have (non-functional), and what it explicitly will not do (out of scope).

Functional requirements
Page management
Create, rename, delete pages
Switch between pages via sidebar
Persist pages across browser sessions
Set an emoji icon per page
Block editing
Type to create paragraph blocks
Enter = new block, Backspace = delete empty
/ command to change block type
Drag handles to reorder blocks
Block types (v1)
Paragraph, Heading 1/2/3
Code block (with language label)
Quote, Divider
Todo (checkbox)
AI generation
/ai command opens prompt input
Claude API generates block JSON
Skeleton loader while waiting
Generated blocks inserted inline
Export
Export page as .md file download
Blocks converted to correct Markdown
Copy to clipboard (nice to have)
UX behaviour
Auto-save on every block change
Keyboard-first navigation
Empty state on new page
Undo/redo (nice to have)
Non-functional requirements
Local-first — all data lives in the browser (IndexedDB). No server, no account, no internet required after first load.
Performance — block edits must feel instant (<16ms). AI skeleton shown within 100ms of submitting prompt.
Type safety — full TypeScript coverage. Block schema is the single source of truth across the whole codebase.
Resilience — a failed AI call never corrupts editor state. Errors shown inline, editor stays usable.
Accessibility — keyboard navigation throughout. Slash menu navigable with arrow keys + Enter.

Architecture

Kiroku is a single-page React app with no backend. The architecture is 5 layers. Click each layer to see its files and responsibilities.

1
UI layer — React components What the user sees and interacts with ▶
Sidebar.tsxPage list, new page button. Reads pagesSlice, dispatches createPage / setActivePage.
EditorCanvas.tsxThe main writing surface. Contains BlockList and SlashMenu, handles keydown events.
Block.tsxSwitch component — reads block.type, renders the correct block component. The block registry lives here.
SlashMenu.tsxFloating command palette. Opens on "/", filters on subsequent keys, closes on Escape or selection.
blocks/One file per block type: ParagraphBlock, HeadingBlock, CodeBlock, TodoBlock, QuoteBlock, DividerBlock.
2
State layer — Redux Toolkit In-memory truth while the app is running ▶
pagesSlice.tsOwns the pages array and activePageId. Actions: createPage, deletePage, renamePage, setActivePage, setPageIcon.
editorSlice.tsOwns blocks for the active page. Actions: addBlock, updateBlock, deleteBlock, moveBlock, insertAIBlocks.
uiSlice.tsOwns transient UI state: slashMenuOpen, slashMenuPosition, aiLoading, sidebarCollapsed. Never persisted.
store/index.tsConfigures the store, wires the StorageMiddleware that auto-saves pagesSlice and editorSlice to IndexedDB on every action.
3
Logic layer — pure services Business logic with no React or Redux dependency ▶
BlockEngine.tscreateBlock(type, content?) factory function. Generates nanoid, sets createdAt, returns a valid Block object.
SlashParser.tsparseSlashCommand(raw: string) — returns { type: 'block' | 'ai', value: BlockType | string }.
AIService.tsgenerateBlocks(prompt: string): Promise<Block[]> — calls Claude API with block schema in system prompt, parses and validates the JSON response.
ExportService.tsexportToMarkdown(blocks: Block[]): string — maps each block type to its Markdown equivalent and joins the result.
4
Storage layer — IndexedDB Persisted data that survives browser refresh ▶
StorageService.tsWraps idb library. Two stores: "pages" (Page objects keyed by id) and "meta" (last active page id). Called by the Redux middleware, never directly by components.
StrategyRedux middleware listens for any pagesSlice or editorSlice action → serialises the full pages array → writes to IndexedDB. On app load, StorageService.loadAll() hydrates the Redux store before first render.
5
External layer — Claude API The only network call in the entire app ▶
EndpointPOST https://api.anthropic.com/v1/messages — called only from AIService.ts, never from components directly.
System promptIncludes the full Block TypeScript interface. Instructs Claude to return only a raw JSON array. No markdown fences, no explanation.
Error handlingNetwork errors, invalid JSON, and schema mismatches are all caught in AIService. On failure, dispatch setAIError(message) — the skeleton block is replaced with an error state, not a crash.

Data model

Two entities. A Page contains an ordered array of Blocks. Everything else in the app is derived from these two shapes.

Page stored in IndexedDB · one per document
idstringnanoid() — primary key
titlestringEditable in the page header, defaults to "Untitled"
iconstringSingle emoji character shown in sidebar
blocksBlock[]Ordered array — index determines visual order
createdAtnumberDate.now() timestamp, for sorting in sidebar
updatedAtnumberUpdated on every block change, shown as "last edited"
Block lives inside Page.blocks · never stored standalone
idstringnanoid() — used as React key and for selection
typeBlockTypeEnum: paragraph | heading_1 | heading_2 | heading_3 | code | quote | todo | divider
contentstringRaw text. Empty string for divider blocks.
propertiesBlockProps?Optional metadata — see BlockProps below
createdAtnumberTimestamp — used for debugging and future sorting
BlockProps all fields optional · only set when relevant to the block type
languagestring?Code blocks only — e.g. "typescript", "python"
checkedboolean?Todo blocks only — whether the checkbox is ticked
aiGeneratedboolean?True if block was created by Claude API — used to show AI badge
Design decisions
Blocks array, not a linked list. Simpler to reorder with index splicing. No pointer chasing for rendering.
Blocks nested inside Page, not a flat table. Kiroku is local-only. A flat table with foreign keys would add join complexity with zero benefit.
content is always a string. Even code blocks. Syntax highlighting is a rendering concern, not a data concern.
No children array on Block. Nested blocks are a v2 problem. Keeping it flat makes the renderer trivial and the AI prompt schema concise.

Interface definitions

The contracts between layers. If these are well-defined, each layer can be built and tested independently.

BlockEngine
createBlock
createBlock(type: BlockType, content?: string, props?: BlockProps): Block
Factory function. Generates a new block with a fresh nanoid, current timestamp, and the provided type/content. This is the only way a Block should be created — never construct Block objects inline.
SlashParser
parseSlashCommand
parseSlashCommand(input: string): SlashResult | null
type SlashResult =
| { kind: 'block'; blockType: BlockType }
| { kind: 'ai'; prompt: string }
Returns null if input doesn't start with "/". Returns a typed result otherwise. The caller (EditorCanvas) decides what to do with the result — this function has no side effects.
AIService
generateBlocks
generateBlocks(prompt: string): Promise<Block[]>
Calls the Claude API. The system prompt includes the Block TypeScript schema. Parses the response JSON, validates each item matches the Block shape, assigns fresh nanoids, and returns the array. Throws a typed KirokuAIError on failure — never returns malformed blocks.
ExportService
exportToMarkdown
exportToMarkdown(blocks: Block[]): string
Pure function — takes a block array, returns a Markdown string. No side effects. The download trigger lives in the UI layer, not here. Mapping: paragraph → plain text, heading_1 → #, code → triple backtick fence, todo → - [ ] or - [x], quote → >, divider → ---.
StorageService
loadAll / saveAll
loadAll(): Promise<Page[]>
saveAll(pages: Page[]): Promise<void>
getLastActivePageId(): Promise<string | null>
setLastActivePageId(id: string): Promise<void>
Thin wrapper over idb. Called only by the Redux middleware and the app bootstrap. No component should ever call StorageService directly — all persistence flows through Redux.

Optimizations

Problems to anticipate and solve before users hit them. Divided into performance, UX, edge cases, and error handling.

Performance
PERF
Debounce IndexedDB writes
Don't write to IndexedDB on every keypress — that's potentially 10 writes/second. Debounce the Redux middleware by 500ms. The UI stays reactive (Redux is synchronous), only the persistence is delayed.
PERF
React.memo on block components
Wrap ParagraphBlock, HeadingBlock etc. in React.memo. When one block updates, only that block re-renders — not the entire BlockList. Critical for pages with 50+ blocks.
PERF
Stable block keys
Always use block.id as the React key — never the array index. Index-based keys cause every block below an insertion to re-mount, destroying focus and cursor position.
UX
UX
Cursor position preservation
After inserting a new block (Enter key), move the cursor to the start of the new block immediately. After deleting a block (Backspace), move cursor to the end of the previous block's text. Use a focusedBlockId ref + useEffect to imperatively focus the right DOM node.
UX
Slash menu filter and positioning
The slash menu must appear at the cursor, not at a fixed position. Use a hidden <span> inside the contenteditable to measure cursor coordinates via getBoundingClientRect(). Filter the menu items as the user types after "/" — fuzzy match on block type names.
UX
AI skeleton block
Insert a placeholder block with type "ai_loading" immediately when the user submits the /ai prompt. This gives instant visual feedback. Replace it with the real blocks once the API resolves — or an error state if it fails.
Edge cases
EDGE
Deleting the last page
If the user deletes the only page, automatically create a new blank "Untitled" page and set it as active. The app should never be in a state with zero pages.
EDGE
Empty page on first load
If IndexedDB has no data (first ever visit), seed the store with one default page containing a single empty paragraph block. Don't show a blank white screen.
EDGE
Paste handling in blocks
By default, pasting into a contenteditable preserves rich HTML formatting, which breaks the block model. Intercept the paste event, extract plain text via e.clipboardData.getData('text/plain'), and insert it manually.
Error handling
ERROR
AI API failure
Replace the ai_loading skeleton block with an "ai_error" block showing the error message and a retry button. Never crash the editor. The user's existing blocks are untouched.
ERROR
Malformed AI JSON response
Claude occasionally returns invalid JSON or JSON that doesn't match the Block schema. Wrap JSON.parse in try/catch. Validate each block against the schema with a type guard before dispatching. Reject the whole batch if any block is invalid.
ERROR
IndexedDB unavailable
IndexedDB can be blocked in private browsing mode on some browsers. Detect the failure in StorageService.init() and fall back gracefully — the app still works, just without persistence. Show a subtle warning banner.
